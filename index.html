Operational Control in the Eye of the Storm

Tornado VPN is a self-hosted secure access platform combining WireGuard performance, Tor-aware routing, and real-time operations visibility. It is designed for organizations that need strict control of security, uptime, and observability without outsourcing network trust.

Architecture

Designed as a Supervised Control Plane with Separated Data Paths

The platform is centered around a master supervisor that starts and monitors service processes defined in runtime config. External access enters through two FastAPI surfaces: client-connect API and admin dashboard API. Internal contracts are standardized over UDS sockets under /run/tornado.

Supervisor Layer

  • `MASTER_service.py` process lifecycle
  • Config-driven service sync/reload
  • Health ping and restart automation

Control Services

  • Auth, session, user, IPAM, key rotator
  • OS/API service managers
  • Log management microservice

Data Plane

  • WireGuard `wg0` and `wg1`
  • Tor manager and relay state control
  • Routing and maintenance-path handling

Persistence

  • Redis: sessions, pools, events, revocation
  • PostgreSQL: users and history
  • SQLite (log service) for query/analytics
Security Model

Multi-Layer Authentication and Key Integrity Controls

Login payloads are protected via X25519 ECDH + HKDF + AES-GCM before credential validation. Token issuance uses asymmetric JWT keypairs with overlap-aware verification during rotations, backed by Redis JTI revocation checks.

Credential Exchange

  • Client requests auth public key
  • Credentials encrypted client-side
  • Server decrypts and forwards to auth service

Token Lifecycle

  • RS256 access and refresh tokens
  • Claim and token-type validation
  • Reuse detection for revoked refresh tokens

Key Management

  • Bootstrap key validator/generator
  • Public-first then private cutover rotation
  • Signal-driven service key reload
Operations Runbook

Production Operations with Deterministic Recovery Paths

Operations are structured around dependency-first checks, targeted service remediation, and post-incident validation. The admin API and logs service provide direct control and forensic visibility for runtime diagnostics.

  • Recovery order: PostgreSQL and Redis first, then `tornado.service`, then NGINX.
  • Session instability triage: heartbeat TTL, keyspace events, and session cleanup telemetry.
  • Rotation incidents: validate overlap key presence and reload signal targets.
  • Tunnel incidents: verify `wg0/wg1`, IP pools, and peer-programming logs.

API Surface

Client and Admin Interfaces for Provisioning, Control, and Telemetry

The client API handles encrypted login, token refresh, tunnel initiation, and heartbeat updates. The admin API exposes service management, user lifecycle operations, logs query/export, metrics streams, tor relay controls, and key-rotator actions.

Client API :4605
  • GET /auth/pubkey
  • POST /auth/login
  • POST /auth/reauth
  • POST /vpn/initiate
  • POST /session/heartbeat
Admin API :8000
  • /services/* and /apps/* controls
  • /users lifecycle and user session ops
  • /logs/* query/tail/export analytics
  • /status /up /down /circuits relay ops
  • /api/key-rotator/* rotation controls
